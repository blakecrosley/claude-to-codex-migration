# Doctrine Migration

Use this reference when moving private philosophy, quality doctrine, taste rules, evidence rules, or operating principles from another harness into Codex.

## Target Shape

Translate doctrine into behavior, not lore.

Use `AGENTS.md` for compact always-on rules:

- The objective function.
- Evidence required before completion.
- Worthiness or taste gates for product and public work.
- Refusal/deletion rules for work that should not exist.
- Escalation rules for high-risk tasks.
- A clear AI-era quality stance: easier code generation raises the value of taste, coherence, restraint, and refusal.

Use a private skill only when the doctrine needs deeper on-demand reasoning. Keep the skill explicit-only until it proves useful in real work.

Use hooks only for deterministic checks. Do not use hooks to inject broad philosophy every turn.

## Migration Method

1. Read the source doctrine privately.
2. Extract operational gates: what the agent must do, refuse, verify, or reconsider.
3. Compress each gate into one plain rule.
4. Place short always-on rules in the closest `AGENTS.md` scope.
5. Put long doctrine, names, examples, and rationale in a private reference or explicit-only private skill.
6. Test on one real task where the doctrine should change the outcome.
7. Promote only the rules that measurably improve behavior.

## Public Boundary

For public posts or public plugins, describe the pattern as "quality doctrine," "evidence gate," "worthiness gate," or "operating principles." Do not publish private doctrine names, raw source text, exact prompt phrasing, or private examples unless the user explicitly approves that disclosure.

## Good Codex Translations

- "Do not claim completion when the user-visible path still fails."
- "Correct work that weakens the product should be simplified, rebuilt, or refused."
- "AI can make code cheaper to produce; quality and taste must become stricter, not looser."
- "Tests support the evidence; they do not replace runtime/user-path verification."
- "Name what remains unverified."

## Poor Codex Translations

- Long motivational essays in always-loaded context.
- A separate profile for every philosophy when the profile does not change mechanics.
- Hooks that inject doctrine without a deterministic pass/fail behavior.
- Public examples that reveal private source material.
